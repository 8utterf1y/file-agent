# 参赛作品提交规范
## 1. 提交内容
参赛作品统一提交至 **work/** 目录，目录结构不限，作品形式不限，包括但不限于：
```markdown
CLI 工具
Agent
Skill
MCP Server
Web 服务
Extension
Docker 应用
其他自动化工具
```
特别说明：本赛题要求工具能够无人值守地完成代码实现与设计文档的一致性检视工作，输入为代码仓及对应的设计文档，输出为代码实现相对设计文档的不一致问题。工具应具有泛化性，可解决不同形式的不一致问题。

## 2. 必须包含文件
参赛作品根目录至少包含：
```markdown
根目录
├── INSTRUCTION.md
└── ...
```

​其中：**INSTRUCTION.md**：裁判执行说明书（详见第 3 节）
Skill 路径要求：如果参赛作品包含 Skill，路径必须为：
```markdown
work/skills/{your-skill-name}/SKILL.md
```

## 3. INSTRUCTION.md 规范
INSTRUCTION.md 为裁判执行说明书，自动评测系统将严格按照其中描述执行参赛作品。
至少应包含以下内容：
### 3.1 环境准备
说明运行参赛作品所需的全部环境准备步骤，包括但不限于：
```markdown
依赖安装
编译构建
服务启动
Docker 镜像准备
MCP、Skill 等组件安装
```
所有步骤应能够自动完成，不得依赖人工操作。

### 3.2 执行流程
说明裁判如何执行参赛作品，包括：
```markdown
启动命令
输入参数说明：
代码仓路径：本地 git 仓路径（基准测试用例位于 code/f-stack，分支为 competition，commit 为 58cc9cf685f496d0542b072fe3e6246d3ceba781）
设计文档路径：代码仓内的设计文档或外部 RFC 文档路径（建议先转为 md 格式）
输入路径
调用方式
执行顺序
```

### 3.3 执行完成判定
说明裁判如何判断作品执行结束，例如：
```markdown
命令退出
服务返回结果
指定文件生成
指定接口返回完成状态
```

### 3.4 结果获取方式
说明裁判如何获取最终执行结果，包括：
```markdown
输出文件路径
接口地址
返回格式
结果格式要求：输出应包含识别出的不一致问题列表，每个问题应提供：
* 不一致问题的描述
* 证据链（设计文档中的规范描述 ↔ 代码实现中的实际行为）
* 涉及的代码位置（文件路径、行号等）
* 涉及的设计文档位置（文档路径、章节等）
```
结果格式不限（JSON、Markdown、CSV 等均可），但应能够支持自动读取。

## 4. 自动评测流程
自动评测系统将按照以下流程执行：
* 阅读 INSTRUCTION.md
* 按照说明完成环境准备
* 按照说明执行参赛作品（输入为 code/f-stack 代码仓及对应 RFC 文档）
* 获取参赛作品输出结果
* 根据赛题标准用例完成自动评分

## 5. 评测标准
### 5.1 基本要求
|指标|要求|
|--|--|
|识别不一致|issues 数量	≥ 4 个（已知存在 6 个 issues，参考 issues 目录，但找出的不限于已知这些）|
|误报率|不高于 50%|
|检视时长|不超过 6 小时|

### 5.2 评优导向
以找出不一致的 issues 数量为主要指标，兼顾误报率及检视时长。

## 6. 平台提供
评测平台会提供项目材料，参赛选手不需要提交这些内容。
平台提供路径如下：
```markdown
/app/code/judge-assets/01_03_ai_implementation_design_difference_detection/
├── code/              # 原始业务代码，参赛作品需要操作的对象
├── Difference/
           ├── benchmark.md   #提供了外部rfc链接地址
```

## 7. 注意事项
以下情况视为执行失败：
```markdown
缺少 INSTRUCTION.md
INSTRUCTION.md 无法完成自动执行
执行过程需要人工交互
无法判断执行结束
无法获取最终执行结果
```
包含 **Skill** 则路径必须符合 **work/skills/{your-skill-name}/SKILL.md** 规范
