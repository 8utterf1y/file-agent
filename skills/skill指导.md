# Agent Skill 编写规范与实践教程

## 摘要

Agent Skill 是一种面向 AI Agent 的能力封装方式。它将某类任务所需的操作流程、领域规则、可执行脚本、参考资料和输出规范组织在一个结构化目录中，使 AI 在处理重复性任务时能够遵循稳定的流程，而不是每次依赖临时提示词重新理解任务。

本文介绍 Skill 的基本概念、典型应用场景、标准目录结构、`SKILL.md` 编写规范、脚本与参考资料的设计方法，以及从需求分析到校验迭代的完整编写流程。本文适用于希望在 Codex、OpenCode、Claude Code 或其他支持 Agent Skills 范式的工具中构建可复用能力包的研发团队。

---

## 1. Skill 的定义

Skill 可以理解为一个面向 AI Agent 的“任务能力包”。它不是单纯的一段提示词，也不是普通插件，而是一个由说明文档、可执行脚本、参考资料和资源模板组成的结构化目录。AI 在遇到匹配任务时，可以加载对应 Skill，并按照其中定义的流程完成任务。

一个最小 Skill 只需要包含一个入口文件：

```text
my-skill/
  SKILL.md
```

当任务复杂度提升时，Skill 可以进一步包含脚本、参考资料和模板资源：

```text
my-skill/
  SKILL.md
  scripts/
  references/
  assets/
```

其中，`SKILL.md` 是 Skill 的入口文件，用于说明 Skill 的适用场景、操作流程、输出要求和安全边界；`scripts/` 用于存放可执行脚本，适合处理格式转换、文件扫描、数据校验等确定性任务；`references/` 用于存放领域规则、字段说明、示例和长文档；`assets/` 用于存放模板、示例文件、图片或其他可复用资源。

---

## 2. Skill 适合解决的问题

Skill 适合沉淀那些反复出现、流程相对固定、需要遵循特定规范的任务。相比每次重新编写提示词，Skill 能够将团队经验、工具调用方式、输出格式和风险边界固化下来，从而提升 AI 执行任务的一致性和可维护性。

典型适用场景包括：

| 场景类型      | 示例任务                            | Skill 的价值                |
| --------- | ------------------------------- | ------------------------ |
| 文档处理      | 从 PDF 中提取表格、检查页码、保留版式信息         | 固化 PDF 渲染、解析、提取和校验流程     |
| Word 文档编辑 | 修改合同格式、处理修订痕迹、添加批注              | 固化 `.docx` 的编辑、验证和格式保持流程 |
| 表格分析      | 分析 Excel 销售数据、生成汇总表和趋势图         | 固化数据清洗、公式生成、图表输出和结果校验    |
| 幻灯片生成     | 根据报告生成 PPT、复刻已有版式、插入图表          | 固化内容组织、版式规则、图表生成和导出流程    |
| 代码审查      | 审查 PR、检查边界条件、识别潜在缺陷             | 固化审查清单、风险分类和反馈格式         |
| 发布说明      | 根据 commit 或 PR 生成 release notes | 固化变更分类、版本说明格式和发布模板       |
| SQL 审查    | 检查慢查询、JOIN 风险、索引使用情况            | 固化 SQL 审查流程、性能风险判断和优化建议  |
| 事故复盘      | 根据日志和时间线生成故障复盘报告                | 固化影响范围、根因分析、处置过程和改进项结构   |

这些任务的共同点是：它们不是一次性闲聊，而是有明确输入、固定流程、可复用规则和稳定输出格式的工作流。

---

## 3. Skill 与 Prompt、Tool、MCP 的关系

在 Agent 系统中，Skill、Prompt、Tool 和 MCP 常被同时提及，但它们关注的层次不同。

| 类型     | 主要形式                   | 适用范围        | 特点               |
| ------ | ---------------------- | ----------- | ---------------- |
| Prompt | 一段自然语言指令               | 临时任务或简单说明   | 轻量，但难以复用和维护      |
| Skill  | `SKILL.md` 加脚本、参考资料和资源 | 重复性工作流和领域任务 | 可复用、可沉淀、可迭代      |
| Tool   | 函数、命令、API 或脚本          | 确定性操作       | 输入输出明确，适合被 AI 调用 |
| MCP    | 标准化工具协议和服务连接方式         | 外部系统集成      | 适合跨系统、跨工具调用      |

从职责上看，Prompt 解决“这一次怎么说”，Skill 解决“以后遇到这类任务都怎么做”，Tool 解决“具体动作如何执行”，MCP 解决“外部系统能力如何标准化接入”。

在实际工程中，一个高质量 Skill 往往会组合 Tool 或脚本。例如 PDF Skill 可以在 `SKILL.md` 中规定处理流程，同时在 `scripts/` 中提供渲染、抽取、合并、压缩或校验脚本。AI 负责理解任务和组织流程，脚本负责执行确定性操作。

## 4. Skill 的标准结构

一个规范的 Skill 目录通常由以下部分组成：

```text
skill-name/
  SKILL.md
  scripts/
  references/
  assets/
  agents/
```

各部分职责如下：

| 组成部分          | 是否必需 | 主要用途                       | 编写建议                 |
| ------------- | ---: | -------------------------- | -------------------- |
| `SKILL.md`    |   必需 | Skill 入口，定义触发条件、流程、边界和输出要求 | 应简洁、明确、面向 AI 执行      |
| `scripts/`    |   可选 | 存放可执行脚本                    | 用于确定性、格式敏感、重复性强的操作   |
| `references/` |   可选 | 存放长规则、字段说明、示例、API 参考       | 仅在需要时由 AI 读取，避免塞入主文件 |
| `assets/`     |   可选 | 存放模板、图片、样例、资源文件            | 适合生成固定格式产物           |
| `agents/`     |   可选 | 存放专用 Agent 配置或展示信息         | 视平台支持情况使用            |

一个 Skill 不应为了显得完整而堆砌文件。原则上，只有当某个文件能直接支持 Skill 的执行、校验或输出质量时，才应纳入 Skill 目录。

---

## 5. `SKILL.md` 的组成

`SKILL.md` 通常由两部分构成：

1. YAML frontmatter；
2. Markdown 正文。

frontmatter 用于告诉 AI 这个 Skill 是什么、什么时候应该使用；正文用于告诉 AI 在 Skill 被激活之后应如何执行任务。

### 5.1 Frontmatter 示例

```markdown
---
name: pdf-processing
description: Use when the user needs to read, extract, merge, split, review, or generate PDF files, especially when layout, page references, tables, or visual fidelity matter.
license: MIT
compatibility: opencode
metadata:
  category: document-processing
  output: structured-report
---
```

### 5.2 字段说明

| 字段              | 说明       | 编写要求                           |
| --------------- | -------- | ------------------------------ |
| `name`          | Skill 名称 | 应简短、稳定、可读，通常使用小写字母和连字符         |
| `description`   | 触发描述     | 应明确任务边界、触发关键词和典型场景             |
| `license`       | 许可证      | 按团队或开源规范填写                     |
| `compatibility` | 兼容平台     | 可标注适配平台，如 `opencode`、`codex` 等 |
| `metadata`      | 扩展信息     | 可包含类别、语言、输出格式、安全级别等            |

其中，`description` 是最关键字段。AI 往往先看到 Skill 的名称和描述，再判断是否加载完整内容。因此，`description` 不应写得过于抽象，也不应写成背景介绍，而应直接描述“何时使用这个 Skill”。

---

## 6. 编写 `description` 的原则

`description` 应该满足四个要求：

1. 明确说明适用任务；
2. 包含用户可能使用的关键词；
3. 明确边界；
4. 避免冗长和泛化。

不推荐：

```yaml
description: A useful document tool.
```

该描述过于泛化，AI 很难判断它是否适用于 PDF、Word、Excel 还是普通文本处理。

推荐：

```yaml
description: Use when the user needs to create, edit, review, or extract content from Word documents (.docx), including formatting preservation, comments, tracked changes, tables, and document structure.
```

该描述明确说明了文件类型、操作类型和关键能力，更利于 AI 判断是否触发。

---

## 7. `SKILL.md` 正文的推荐结构

`SKILL.md` 的正文应以执行任务为中心，而不是写成项目介绍或背景材料。推荐结构如下：

```markdown
# Skill Name

## Purpose

## When to use

## When not to use

## Inputs and outputs

## Standard workflow

## Safety and approval rules

## Output contract

## Error handling

## References
```

如果团队主要使用中文，也可以使用中文标题：

```markdown
# Skill 名称

## 目标

## 适用场景

## 不适用场景

## 输入与输出

## 标准工作流

## 安全边界与审批规则

## 输出格式

## 异常处理

## 参考资料
```

正文应尽量使用明确的步骤、检查项、表格和示例，避免“全面分析”“专业处理”“合理优化”这类抽象表达。AI 需要的是可执行流程，而不是原则口号。

## 8. 编写 Skill 的标准流程

### 8.1 第一步：明确任务边界

在编写 Skill 之前，应先回答以下问题：

| 问题              | 说明                              |
| --------------- | ------------------------------- |
| 这个 Skill 解决什么问题 | 例如处理 PDF、生成 PPT、审查 SQL、分析 Excel |
| 用户通常如何表达需求      | 收集自然语言触发句                       |
| 输入是什么           | 文件、目录、表格、代码、日志、API 响应等          |
| 输出是什么           | 报告、文件、结构化 JSON、代码补丁、图表等         |
| 是否需要脚本          | 若任务需要确定性处理，应提供脚本                |
| 是否涉及高风险操作       | 如删除文件、修改文档、调用外部系统、发送消息等         |
| 是否需要人工确认        | 高风险操作应默认要求确认或 dry-run           |

这一阶段的目标不是写文件，而是收敛 Skill 的能力边界。边界越清晰，Skill 越稳定。

---

### 8.2 第二步：收集具体任务样例

Skill 的设计不应从抽象定义开始，而应从真实任务样例开始。具体样例能帮助作者判断 Skill 需要哪些脚本、规则、输入输出和异常处理。

以下是不同类型 Skill 的样例。

| Skill 类型              | 用户请求示例                           | 设计启示                   |
| --------------------- | -------------------------------- | ---------------------- |
| PDF Skill             | “请提取这份 PDF 第 5 页的表格，并保留页码来源。”    | 需要表格提取、页码引用、版式保持       |
| DOCX Skill            | “请把这份合同里的批注整理出来，并生成修订摘要。”        | 需要读取批注、修订、段落结构         |
| XLSX Skill            | “请分析这个 Excel 的销售数据，按地区生成汇总和趋势图。” | 需要表格读取、分组统计、图表输出       |
| PPTX Skill            | “请根据这份研究报告生成一套 10 页演示文稿。”        | 需要内容提炼、版式规划、图表插入       |
| Code Review Skill     | “请审查这个 PR 是否存在边界条件和测试不足。”        | 需要 diff 分析、风险分类、测试覆盖判断 |
| Release Notes Skill   | “请根据这些 commit 生成版本发布说明。”         | 需要变更归类、用户影响分析、版本格式     |
| Incident Report Skill | “请根据这些日志和聊天记录生成故障复盘。”            | 需要时间线、影响范围、根因和改进项      |
| SQL Review Skill      | “请检查这条 SQL 是否有性能风险。”             | 需要 JOIN、过滤条件、索引和执行计划分析 |

通过样例可以判断：哪些内容应写进 `SKILL.md`，哪些应作为 `references/`，哪些必须写成脚本。

---

### 8.3 第三步：规划 Skill 资源

根据任务样例，决定需要哪些资源。

| 资源类型          | 适合放什么      | 示例                               |
| ------------- | ---------- | -------------------------------- |
| `SKILL.md`    | 核心工作流和安全边界 | “先解析文件，再检查输出，再生成报告”              |
| `scripts/`    | 稳定可执行操作    | PDF 渲染、Excel 读取、Markdown 转换、代码扫描 |
| `references/` | 长规则和示例     | 字段定义、报告模板、审查清单、风险分类              |
| `assets/`     | 模板和示例资源    | PPT 模板、报告模板、品牌样式、示例表格            |

判断原则是：如果某段内容每次都需要 AI 理解并遵守，就写入 `SKILL.md`；如果内容较长、只有特定场景才需要，就放入 `references/`；如果任务需要稳定执行某个动作，就放入 `scripts/`；如果任务需要使用模板或素材，就放入 `assets/`。

---

### 8.4 第四步：编写 `SKILL.md`

编写 `SKILL.md` 时，应遵循以下原则：

| 原则       | 说明                   |
| -------- | -------------------- |
| 面向 AI 执行 | 不写冗长背景，直接说明怎么做       |
| 使用明确步骤   | 让 AI 知道第一步、第二步、第三步   |
| 明确输出格式   | 指定表格、JSON、报告或文件结构    |
| 明确安全边界   | 说明哪些操作不能自动执行         |
| 明确异常处理   | 说明低覆盖、解析失败、输入不足时怎么回答 |
| 指向参考资料   | 不把所有细节堆进正文           |

---

### 8.5 第五步：实现脚本

脚本适合承担确定性任务。以不同 Skill 为例：

| Skill 类型            | 适合脚本完成的任务                   |
| ------------------- | --------------------------- |
| PDF Skill           | 渲染页面、提取文本、合并拆分 PDF、检测空白页    |
| DOCX Skill          | 解包 `.docx`、处理 XML、提取批注、接受修订 |
| XLSX Skill          | 读取表格、计算统计值、生成图表、校验公式        |
| PPTX Skill          | 生成幻灯片、插入图表、检测文字溢出           |
| Code Review Skill   | 收集 diff、统计变更文件、运行测试或 lint   |
| SQL Review Skill    | 格式化 SQL、抽取表名、调用 explain 工具  |
| Release Notes Skill | 读取 commit、按类型归类、生成模板化说明     |

脚本不应承担需要业务判断的任务。例如，脚本可以告诉 AI 哪些函数相似，但是否应该合并，需要结合调用关系、业务语义、测试覆盖和人工确认。

---

### 8.6 第六步：编写参考资料

当规则过多时，应放入 `references/`。

常见参考资料包括：

```text
references/
  output-schema.md
  examples.md
  risk-rules.md
  troubleshooting.md
```

示例：

| 文件                   | 内容                    |
| -------------------- | --------------------- |
| `output-schema.md`   | 输出字段、JSON Schema、报告结构 |
| `examples.md`        | 典型用户请求、正确回答、反例        |
| `risk-rules.md`      | 风险等级、人工审批规则、高风险操作说明   |
| `troubleshooting.md` | 常见失败原因和处理方式           |

参考资料应服务于 Skill 执行，不应变成独立教程或无关背景材料。

---

### 8.7 第七步：验证 Skill

一个 Skill 至少应经过三类验证：

| 验证类型 | 验证内容                                 |
| ---- | ------------------------------------ |
| 格式验证 | `SKILL.md` frontmatter 是否合法，目录结构是否正确 |
| 功能验证 | 脚本是否可运行，输出文件是否符合预期                   |
| 行为验证 | AI 是否按 Skill 流程执行，是否遵守安全边界和输出格式      |

例如，对于 PDF Skill，需要验证 PDF 渲染脚本是否能处理多页文档；对于 Excel Skill，需要验证公式和图表是否正确；对于代码审查 Skill，需要验证 AI 是否会先阅读 diff，而不是直接输出泛泛建议。

---

### 8.8 第八步：根据真实使用迭代

Skill 不应一次性追求完美。更合理的方式是先实现最小可用版本，再根据真实对话修正：

| 阶段 | 目标                   |
| -- | -------------------- |
| V0 | 只有 `SKILL.md`，定义清楚流程 |
| V1 | 加入脚本，完成确定性处理         |
| V2 | 加入参考资料，完善规则和示例       |
| V3 | 加入测试和校验命令            |
| V4 | 加入更严格输出 Schema       |
| V5 | 引入平台工具包装或自动化调用       |
| V6 | 增加人工确认后的写入或修改流程      |

## 9. 不同类型 Skill 的设计示例

### 9.1 PDF 处理 Skill

适用场景包括阅读、提取、合并、拆分、审查和生成 PDF。该类 Skill 的关键点是处理版式和页码来源，因为 PDF 中的图表、页眉、页脚和多栏布局可能影响理解结果。

推荐能力边界：

| 项目   | 设计建议                   |
| ---- | ---------------------- |
| 输入   | PDF 文件                 |
| 输出   | 文本、表格、图片、页码引用、处理后的 PDF |
| 脚本   | 页面渲染、文本抽取、表格抽取、合并拆分    |
| 风险   | 纯文本抽取可能丢失版式，复杂表格需要视觉核对 |
| 输出要求 | 必须保留页码和证据来源            |

---

### 9.2 DOCX 文档处理 Skill

适用场景包括创建、编辑、审查和格式化 Word 文档。该类 Skill 的关键点是保留文档结构、样式、批注和修订痕迹。

推荐能力边界：

| 项目   | 设计建议                   |
| ---- | ---------------------- |
| 输入   | `.docx` 文件或文档内容        |
| 输出   | 修改后的 `.docx`、审查报告、批注摘要 |
| 脚本   | 文档解包、XML 修改、批注提取、格式检查  |
| 风险   | 直接改 XML 容易破坏文件结构，需要校验  |
| 输出要求 | 保留格式、段落结构和修订说明         |

---

### 9.3 XLSX 表格分析 Skill

适用场景包括数据清洗、统计分析、透视汇总、公式生成和图表制作。该类 Skill 的关键点是保证计算结果准确，并明确数据来源和处理规则。

推荐能力边界：

| 项目   | 设计建议                 |
| ---- | -------------------- |
| 输入   | `.xlsx`、`.csv` 或表格数据 |
| 输出   | 分析结果、图表、汇总表、修改后的工作簿  |
| 脚本   | 表格读取、数据清洗、公式校验、图表生成  |
| 风险   | 错误公式或错误聚合会影响结论       |
| 输出要求 | 说明清洗规则、统计口径和异常数据处理方式 |

---

### 9.4 PPTX 幻灯片生成 Skill

适用场景包括创建演示文稿、复刻模板、插入图表和优化排版。该类 Skill 的关键点是结构、版式和可编辑性。

推荐能力边界：

| 项目   | 设计建议                    |
| ---- | ----------------------- |
| 输入   | 文档、报告、提纲、图片或模板          |
| 输出   | `.pptx` 文件和必要的生成脚本      |
| 脚本   | 生成 PPT、插入图表、检测文字溢出、导出预览 |
| 风险   | 内容可能正确但版式不可用，需要视觉检查     |
| 输出要求 | 保持可编辑，避免只输出截图式幻灯片       |

---

### 9.5 Code Review Skill

适用场景包括 PR 审查、风险识别、测试覆盖检查和可维护性评估。该类 Skill 的关键点不是泛泛评价代码质量，而是基于 diff 和项目上下文进行证据化审查。

推荐能力边界：

| 项目   | 设计建议                      |
| ---- | ------------------------- |
| 输入   | diff、PR、代码文件、测试结果         |
| 输出   | 审查意见、风险等级、修改建议            |
| 脚本   | 收集 diff、运行测试、运行 lint、统计变更 |
| 风险   | 脱离上下文的审查容易误判              |
| 输出要求 | 每条问题必须包含文件、行号、证据和建议       |

---

## 10. 安全边界与审批规则

凡是可能修改文件、发送消息、调用外部系统、删除资源或改变线上状态的 Skill，都必须明确安全边界。推荐将 Skill 分为两个阶段：

| 阶段   | 默认行为                     | 是否需要确认 |
| ---- | ------------------------ | -----: |
| 分析阶段 | 读取文件、运行脚本、生成报告           |  通常不需要 |
| 修改阶段 | 改文件、生成 patch、移动文件、调用外部系统 | 需要明确授权 |

例如，在文档处理 Skill 中，提取 PDF 文本通常可以直接执行；但覆盖原 PDF、发送报告邮件或修改正式合同，应要求用户确认。在代码类 Skill 中，分析重复代码可以默认执行；但真正重构、删除文件或修改公共接口，必须进入单独的 patch workflow。

## 11. Skill 编写质量检查清单

| 检查项                | 说明                          |
| ------------------ | --------------------------- |
| `SKILL.md` 是否存在    | Skill 必须有入口文件               |
| frontmatter 是否有效   | 必须包含 `name` 和 `description` |
| `description` 是否具体 | 是否包含触发场景和边界                 |
| 正文是否面向 AI 执行       | 是否是流程，而不是背景文章               |
| 是否明确适用场景           | 用户什么时候应该使用                  |
| 是否明确不适用场景          | 防止 Skill 被错误触发              |
| 是否有标准工作流           | AI 是否知道执行顺序                 |
| 是否有输出契约            | 输出是否稳定、可评估                  |
| 是否有安全边界            | 是否说明高风险动作如何处理               |
| 是否合理使用脚本           | 确定性任务是否由脚本完成                |
| 是否合理使用参考资料         | 长规则是否放入 `references/`       |
| 是否有测试或校验命令         | 能否验证 Skill 可用               |
| 是否有异常处理            | 输入不足、解析失败、低覆盖时如何处理          |
| 是否可迭代              | 是否便于后续扩展                    |

## 12. 常见问题

### 12.1 Skill 是否越详细越好

不是。Skill 应该足够明确，但不应臃肿。`SKILL.md` 应保留核心工作流和边界，详细规则放入 `references/`，确定性操作交给 `scripts/`。

### 12.2 是否所有任务都需要写 Skill

不需要。一次性任务、开放式讨论、简单问答不适合写 Skill。Skill 更适合重复、高频、流程稳定、需要一致输出的任务。

### 12.3 Skill 是否应该直接修改文件

这取决于任务风险。低风险、可校验、用户明确要求的修改可以进入修改流程；但默认应采用分析优先、dry-run 优先、确认后执行的策略。

### 12.4 Skill 是否需要脚本

如果任务需要稳定计算、格式转换、批量处理或校验，建议提供脚本。如果任务主要是写作、总结、判断和解释，则可以先只写 `SKILL.md` 和参考资料。

### 12.5 Skill 是否需要 README

如果 Skill 仅供 AI 使用，`SKILL.md` 是核心。若团队成员需要安装、维护、调试或复用该 Skill，可以提供简洁的 `README.md`。README 面向人，`SKILL.md` 面向 AI。

---

## 13. 总结

Skill 是一种将任务流程、领域规则、脚本能力和输出规范封装为可复用能力包的方法。它的价值不在于写更多文字，而在于让 AI 在特定任务中更稳定、更可控、更可验证。

一个高质量 Skill 应具备以下特征：

| 特征   | 说明                      |
| ---- | ----------------------- |
| 触发明确 | `description` 能准确匹配用户意图 |
| 流程清晰 | AI 知道该按什么顺序执行           |
| 输出稳定 | 报告、表格或文件格式可预测           |
| 边界清楚 | 高风险操作不会被默认执行            |
| 脚本可靠 | 确定性操作由脚本完成              |
| 资料分层 | 长规则和示例按需读取              |
| 可验证  | 有测试、校验命令或样例             |
| 可迭代  | 能根据真实使用反馈持续优化           |

对团队而言，Skill 可以把个人经验沉淀为团队资产。一个设计良好的 Skill 能减少重复提示、降低执行偏差，并让 Agent 在复杂任务中遵循稳定、可审计的工作流。
