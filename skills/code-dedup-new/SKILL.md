# Code Dedup

## 目标

扫描代码仓库或指定目录，发现重复文件、重复函数、重复类、重复配置、重复脚本和近重复代码块。脚本生成确定性证据，执行者基于证据继续读取源码、检查上下文，并按中文对话习惯给出试运行性质的合并/保留建议。

## 何时使用

- 用户要求查找重复代码、近重复代码或复制粘贴式重复。
- 用户要求分析重复函数、类、模块、配置、脚本或目录级重复。
- 用户要求给出合并建议、抽公共函数、抽公共模块、保留说明或风险分析。
- 用户质疑“为什么没扫到子文件夹”或“为什么没有合并建议”。

## 不适用场景

- 图片、音视频、模型权重、压缩包等二进制资产去重。
- 数据集清理、文档治理、运行时性能分析、安全漏洞扫描。
- 自动修改、删除、移动、重命名、覆盖文件。
- 接入服务、数据库、后台任务、LLM API 或复杂依赖。

## 分析模式

本 Skill 提供三种运行模式：

| 模式 | 适用场景 | 成本 |
|---|---|---:|
| `exact` | 快速检查完全重复文件、完全重复函数/类、完全重复配置 | 低 |
| `standard` | 默认模式，常规代码重复检测、近重复函数/配置分析和合并建议 | 中 |
| `deep` | 用户明确要求完整近重复分析或接受较高成本时使用 | 高 |

默认使用 `standard`。

- 用户说“快速”“先看看”“只查完全重复”“不要近重复”时，使用 `exact`。
- 用户说“扫描重复代码”“给合并建议”“检查当前项目”“分析目录重复”时，使用 `standard`。
- 用户说“完整扫描”“深度分析”“不要漏掉近重复”“可以跑久一点”“低优先级也展开”时，使用 `deep`。

## 标准工作流

1. 先判断用户意图和扫描范围。
2. 如果用户只是询问用法、解释已有报告、询问字段含义、询问为什么结果为 0，或分析某个重复簇，不要重新运行完整扫描。
3. 如果用户要求扫描，再根据意图选择 `exact`、`standard` 或 `deep`。
4. 运行对应模式生成 JSON 和 Markdown 报告。
5. 先查看扫描概览、质量预警和重复项汇总。
6. 只对高/中优先级或用户关注的重复簇继续读取源码。
7. 合并建议必须检查导入、引用、调用方、模块边界、测试覆盖、配置环境、公共 API 和安全/权限/部署影响。
8. 用中文输出结论、证据表、差异分析、风险和试运行建议，并说明当前未修改任何文件。
9. 默认只读分析，不修改文件，不生成补丁。

脚本示例：

```bash
python skills/code-dedup-new/scripts/code_dedup.py --root . --mode exact --output .opencode/reports
python skills/code-dedup-new/scripts/code_dedup.py --root . --mode standard --output .opencode/reports
python skills/code-dedup-new/scripts/code_dedup.py --root . --mode deep --output .opencode/reports
```

范围指令映射：

- “检查当前项目/整个仓库/全项目”：使用 `--root .`，不传 `--include`。
- “检查此目录/当前目录”：如果当前工作目录就是目标目录，使用 `--root .`；如果用户指的是当前文件所在目录，先确认或推断该目录，再用 `--root <目录>`。
- “检查某个目录下的重复项”：使用 `--root . --include <目录>`；如果该目录是独立项目根目录，也可以使用 `--root <目录>`。
- “检查两个目录/项目目录下的重复项”：使用 `--root <共同父目录> --include <目录A> <目录B>`；如果路径不在同一父目录，可传绝对路径或 `../` 相对路径。
- “只比较两个目录之间是否重复”：仍用 `--include <目录A> <目录B>` 先生成证据，再优先查看 `directory_summary.folder_pairs` 和跨目录重复簇。

## 输出要求

除非用户明确要求英文，最终回答必须使用中文，至少包含：

1. 扫描概览。
2. 重复项汇总表。
3. 高优先级重复项详情；如果没有高优先级，则展示最值得复核的中优先级重复项。
4. 每个重复项的证据表。
5. 相似点与差异点分析。
6. 调用关系 / 引用关系检查状态。
7. 合并或保留建议。
8. 风险和阻断点。
9. 验证方式。
10. 当前未修改任何文件的说明。

## 证据复核要求

脚本报告只是候选证据。执行者必须结合源码判断：

- 是否是实际重复，还是样板代码、生成代码、测试夹具或配置格式相似。
- 差异是否来自业务域、权限、异常处理、输入校验、环境配置或边界条件。
- 导入、引用和调用方是否会受影响。
- 是否涉及公共 API、安全、支付、权限、部署或生产配置。

当用户询问某个重复簇能否合并时，读取已有 JSON 报告，定位该重复簇，只读取它涉及的源码并搜索导入、引用和调用方；除非用户明确要求重新扫描，否则不要重跑全仓库。

## 合并建议要求

- 近重复不等于语义等价，合并建议必须经过人工审核。
- 所有建议项必须包含 `requires_manual_review=true`、`dry_run_only=true`、`risk`、`blockers`、`validation`。
- 不允许只因为文件重复就建议删除。
- 不允许只因为函数相似就建议抽公共函数。
- 近重复默认要求人工复核。
- `config_duplicate` 默认高风险。
- 跨模块重复默认谨慎处理。
- 精确重复也只能建议候选去重、归档或合并前复核，不能直接建议删除。

合并建议必须考虑：

- 导入、引用和调用方。
- 是否被外部模块引用。
- 是否属于不同业务域。
- 是否是测试、夹具或示例代码。
- 是否是配置环境差异。
- 是否有测试覆盖。
- 是否会改变公共 API。
- 是否涉及安全、支付、权限、部署、生产配置。

## 低覆盖 / 零结果处理

当 `duplicate_clusters=0` 或扫描文件很少时，不要直接说“项目没有重复代码”。必须解释：

- 实际扫描根目录和 include/exclude 参数。
- 递归看到的子目录数量。
- 命中可分析代码的目录。
- 不支持的扩展名和被忽略文件。
- `quality_warnings` 是否提示低覆盖、比较上限或测试目录限定。

必要时用 `rg --files <目录>` 复核文件列表，并调整参数重跑。

## 后续补丁/重构工作流

如果用户后续明确批准某条建议，可以进入单独的补丁/重构工作流。该流程必须重新读取受影响文件，检查导入、引用、调用方和测试，生成最小补丁计划；中/高风险改动必须再次确认；用户确认后才进入实际修改，修改后运行可用测试，并保持改动范围窄。

本 Skill 的默认扫描/报告阶段不实现补丁。
