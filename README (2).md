## 题目名称
识别代码实现和设计文档中的不一致问题

## 题目背景
代码实现和架构设计、软件实现设计一致问题一直我司产品质量可信的一个重要课题。在人手写代码时代，经常由于赶工、业务理解不透彻等原因导致代码实现和设计文档有差异；AI coding的新时代，以spec为核心的SDD开发流程正成为主流，但由于大模型漂移经常会出现一些AI生成的代码和原有的SPEC不一致的情形。
为此，我们希望有一套智能的一致性检视系统，尽可能多地识别出实现和设计的差异，同时保证一定的准确率。


## 验证工程
1. 验证工程已下载到[code/f-stack](./code/f-stack)
2. 请基于相关的RFC文档(参见[benchmark](./Difference/benchmark.md))分析。
3. 已知存在一些不一致([issues清单](./Difference/issues/00-summary.md))。
4. 出题方会使用其他存在不一致问题的内部工程项目进行能力评测。
蓝区参赛的选手可以通过开源地址下载：
```
https://github.com/F-Stack/f-stack
```
下载后切换到指定分支
```
git checkout -b competition 58cc9cf685f496d0542b072fe3e6246d3ceba781
```

## 解题要求
1. 工具应尽可能无人值守地完成检视工作。
2. 工具具有泛化性，可解决不同形式的不一致问题。
3. 可复用公司内外的工具，使用公司内的算力资源。

## 交付件要求
1. 交付件形式：
(1) 工具：可以是独立的CLI或Web服务工具，或者extension、agent、command、skill等等；输入为代码仓（本地git仓或codehub仓链接）及对应的设计文档（可以是代码仓内的设计文档、外部RFC文档等等；建议先转成md格式），输出为代码实现相对设计文档的不一致问题。
(2) 文档：技术报告，包含技术方案、工具部署及使用手册、在基准测试用例上的效果验证等内容。
2. 验收标准：
(1) 针对基准测试用例(入围标准): 工具应正确识别基准测试用例中存在的不一致issues>=4个(已知存在6个issues，参考[issues目录](./issues/)，但找出的不一致issues不限于已知的这些)，并给出不一致的证据链;
(2) 误报率不高于50%; 
(3) 检视问题时长不超过6小时。
3. 评优标准：
出题方根据参赛人提供的工具及文档进行本地部署并验证;以找出不一致的issues数量为主要指标，兼顾误报率及检视时长的作品将获得更高主观评分。
